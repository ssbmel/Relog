# 공통 개발 규칙

이 문서는 프로젝트 작업 시 **항상** 준수해야 하는 핵심 규칙들을 정의합니다.

---

## 🔒 보안 규칙 (Security Rules)

### 환경 변수 파일 접근 금지

- `.env`, `.env.local`, `.env.production` 등 **모든 환경 변수 파일에 절대 접근 금지**
- 사용자가 명시적으로 요청하더라도 접근하지 않음
- 환경 변수 관련 질문에는 일반적인 가이드만 제공

### 보안 위험 사전 평가

- 모든 작업 수행 전, 보안적 위험성을 먼저 판단
- 애매하거나 위험 가능성이 있는 경우 **반드시 사용자 동의 구하기**
- 다음 작업은 특히 주의:
  - 데이터베이스 스키마 변경
  - 인증/권한 관련 코드 수정
  - 외부 API 키 사용
  - 파일 시스템 접근

### 로직 보안 검토

- 구현한 로직에 보안 문제가 있다고 판단되면 **즉시 사용자에게 알림**
- 잠재적 취약점 (예: SQL Injection, XSS, CSRF 등) 사전 차단

## 📝 파일 및 코드 수정 규칙

### 명시되지 않은 페이지 수정 금지

- 사용자가 **명시적으로 요청한 파일/페이지만** 수정
- 연관 파일 수정이 필요한 경우, 먼저 사용자에게 설명하고 동의 구하기
- 예시:
  ```
  ❌ "이 기능을 위해 A, B, C 파일을 수정했습니다"
  ✅ "이 기능을 구현하려면 A, B, C 파일 수정이 필요합니다. 진행할까요?"
  ```

## 📦 라이브러리 및 의존성 관리

### 라이브러리 설치 금지

- 사용자가 명시적으로 요청하지 않은 라이브러리는 **절대 설치 금지**
- 새 라이브러리가 필요한 경우:
  1. 사용자에게 필요성 설명
  2. 대안 제시 (기존 라이브러리 활용 가능 여부)
  3. 동의 후 설치

### package.json 사전 분석 필수

- **모든 개발 작업 전** `package.json` 확인
- 다음 사항 분석:
  - 프로젝트에서 사용 가능한 라이브러리 목록
  - 각 라이브러리의 버전
  - 이미 설치된 라이브러리로 해결 가능한지 우선 검토
- 예시:

  ```typescript
  // ❌ 새 라이브러리 설치 제안
  // "date-fns를 설치해서 날짜 포맷팅 하겠습니다"

  // ✅ 기존 라이브러리 활용
  // "package.json을 확인한 결과 이미 설치된 dayjs로 구현 가능합니다"
  ```

## 🎨 Figma 디자인 구현

### Figma 링크 제공 시 프로세스

1. **전체 Figma 구조 분석 먼저 수행**
   - 모든 페이지/프레임 파악
   - 컴포넌트 계층 구조 이해
   - 디자인 시스템 (색상, 타이포그래피, 간격 등) 추출

2. **Step-by-Step 구현**
   - 한 번에 모든 것을 구현하지 않음
   - 각 단계별로 사용자 확인 후 진행
   - 구현 순서:
     1. 디자인 토큰 정의 (CSS 변수)
     2. 공통 컴포넌트
     3. 레이아웃 구조
     4. 개별 페이지

## ⚛️ Next.js App Router (v16+) 규칙

### 프로젝트 구조 인식

- 이 프로젝트는 **Next.js App Router (v16+)** 를 사용
- Pages Router 패턴 사용 금지

### 작업 전 필수 분석 (Step-by-Step)

모든 Next.js 작업 전 다음을 **순차적으로** 분석:

1. **폴더 구조 분석**

   ```
   app/
   ├── (routes)/
   ├── api/
   └── layout.tsx
   ```

   - `app` 디렉토리 기반 라우팅 확인
   - Route Groups `()` 사용 여부 파악
   - 병렬 라우트 `@folder` 사용 여부 파악

2. **라우터 구조 분석**
   - 동적 라우트 `[id]` 패턴 확인
   - Catch-all 라우트 `[...slug]` 확인
   - 인터셉팅 라우트 `(..)` 확인

3. **HTML/CSS 뼈대 및 Layout 구조 분석**
   - 루트 `layout.tsx` 구조 파악
   - 중첩 레이아웃 계층 이해
   - 메타데이터 설정 방식 확인
   - 서버/클라이언트 컴포넌트 구분 확인

### Next.js 16+ 특화 사항

- Server Components가 기본값
- `'use client'` 명시적 선언 필요 시에만 사용
- Server Actions 활용 (`'use server'`)
- `async` 컴포넌트 지원 활용

## ✅ 작업 완료 후 품질 보증

### 전체 검토 프로세스 (Step-by-Step)

모든 작업 완료 후 **반드시** 다음 순서로 검토:

1. **기능 완성도 검토**
   - 요청된 모든 기능이 구현되었는가?
   - 빠진 부분이 없는가?

2. **코드 품질 검토**
   - TypeScript 타입 에러 없는가?
   - ESLint 경고 없는가?
   - 불필요한 코드 제거했는가?

3. **디테일 수정**
   - UI/UX 개선 가능한 부분
   - 성능 최적화 가능한 부분
   - 접근성 (a11y) 개선 가능한 부분

4. **보안 재검토**
   - 민감한 데이터 노출 위험 없는가?
   - 인증/권한 체크 누락 없는가?

### 체크리스트 반환 필수

작업 완료 시 **항상** 다음 형식의 체크리스트 제공:

```markdown
## 구현 완료 체크리스트

### ✅ 완료된 작업

- [ ] 기능 A 구현
- [ ] 기능 B 구현
- [ ] 타입 정의

### 🔍 검토 완료 항목

- [ ] TypeScript 타입 에러 없음
- [ ] ESLint 경고 없음
- [ ] 보안 검토 완료

### 📝 추가 고려사항

- 향후 개선 가능한 부분
- 알려진 제한사항
```
